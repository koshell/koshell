# Design 0003 â€” `shell-init` auto-wrap snippet

Date: 2026-07-03 10:26:47 CST

## Why

Users want koshell to take over every new terminal transparently, without remembering
to type `koshell` first. The obvious mechanism is an `exec koshell` line in the shell
rc file, but a hand-written line hard-codes guard logic (recursion, TTY checks, escape
hatches) into user dotfiles where koshell can never fix or evolve it. Following the
starship/zoxide precedent, koshell now owns the snippet behind a subcommand:

```sh
# top of ~/.zshrc
eval "$(koshell shell-init zsh)"

# top of ~/.bashrc (on macOS, ensure ~/.bash_profile sources ~/.bashrc)
eval "$(koshell shell-init bash)"
```

## Why `exec` works with the existing design

Two pre-existing pieces make the exec approach safe rather than clever:

- **The `KOSHELL=1` marker** (`crates/koshell-rs/src/shell.rs`) is forced into the
  child PTY environment, so the snippet can use it to break recursion.
- **The integration rc re-sources the user's rc** (`crates/koshell-rs/src/
shell_integration.rs`): bash via `--rcfile` (sourcing `~/.bashrc`), zsh via a
  temporary `ZDOTDIR` (sourcing `~/.zshenv`, `~/.zprofile`, `~/.zshrc`). The inner
  shell therefore runs the same snippet, sees the marker, and skips the `exec` while
  the rest of the user's configuration loads normally.

Because the snippet sits at the top of the rc file, the outer shell is replaced before
it does any expensive initialization; the full rc runs exactly once, in the wrapped
shell.

## The snippet

Generated by `koshell shell-init {bash,zsh}` (`crates/koshell-rs/src/shell_init.rs`).
Guards, in order:

- interactive shell only (`$- == *i*` for bash, `[[ -o interactive ]]` for zsh) â€”
  keeps scripts and `bash -c` unaffected even if the rc is sourced from odd places;
- not already inside koshell (`KOSHELL` unset);
- escape hatches: `KOSHELL_NO_AUTO=1` in the environment, or the existence of
  `${XDG_CONFIG_HOME:-$HOME/.config}/koshell/no-auto`. The file form exists because a
  koshell binary that crashes at startup closes the terminal window immediately after
  `exec`; a flag file can be created without a working interactive shell (another
  terminal app, SSH, Finder, or a terminal profile running `zsh -f`);
- stdin and stdout are TTYs (`-t 0 && -t 1`) â€” excludes `ssh host cmd`, scp, and IDE
  background shells;
- `TERM != dumb` â€” excludes Emacs TRAMP and similar;
- `command -v koshell` â€” never `exec` a missing binary.

The snippet is `eval`-safe: the guard path falls through without `exit`/`return`, so a
shell that fails the guard continues its rc unaffected.

## File placement guidance

- zsh: top of `~/.zshrc`. Not `~/.zshenv` (runs for non-interactive shells too), not
  `~/.zprofile` (login shells only; Linux terminals often start non-login shells).
- bash: top of `~/.bashrc`, with `~/.bash_profile` sourcing `~/.bashrc` (macOS
  terminals start login shells that read only `~/.bash_profile`). koshell's generated
  bash rcfile sources `~/.bashrc`, so the guard must live there to be re-evaluated in
  the inner shell.

## CLI integration

The flat `trailing_var_arg` CLI became a clap subcommand enum
(`crates/koshell-rs/src/cli.rs`): `shell-init` is a real subcommand, and the direct
program launch (`koshell python3 -i`) moved to an `external_subcommand` variant.
Verified preserved semantics: dashed arguments after the program pass through,
`--log-level` before the program is koshell's, `--` still allows dashed program names,
unknown dashed options are rejected. Accepted residual: a program literally named
`shell-init` now needs a path spelling (`koshell ./shell-init`).

## Tests

- `crates/koshell-rs/src/shell_init.rs` â€” snippet syntax checks via `bash -n` /
  `zsh -n` (skip when the interpreter is missing) and guard-presence assertions.
- `crates/koshell-rs/src/cli.rs` â€” subcommand parsing plus regression tests for the
  pre-existing launch semantics.
- `crates/koshell-rs/tests/shell_init_pty.rs` â€” end-to-end: spawns the real shell in a
  PTY with the built `koshell` on `PATH` and the `eval` line in an isolated rc, then
  asserts the probe answers from inside the wrap (`KOSHELL=1`), and that
  `KOSHELL_NO_AUTO=1` leaves the original shell in place.

## Open issues

- **Startup-crash lock-out UX**: if koshell starts and dies before the PTY session is
  usable, the window closes too fast to read the error. The escape hatches mitigate;
  a friendlier option would be a "hold and show the error for N seconds when launched
  via auto-wrap" mode (the snippet could pass a flag once the option namespace grows).
- **tmux topology**: tmux started outside koshell gets one koshell per pane
  (auto-wrap in each pane shell); tmux started inside koshell inherits `KOSHELL=1`
  and its panes stay unwrapped. Both are coherent, but the difference is undocumented
  for users beyond this note.
- **`SHLVL` off by one** in the wrapped shell. Cosmetic; no action planned.
- The no-auto flag file is checked only by the snippet, not by koshell itself;
  running `koshell` manually ignores it. Intentional for now.
